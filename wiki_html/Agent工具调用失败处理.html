Agent 工具调用失败处理
从 try/except 到让模型自己想办法

From try/except to Agent-Adaptive Failure Handling:
Why the Old Way Breaks and What Replaces It

LLM-WIKI · 精品感悟  |  2026年5月 · 融合 OpenClaw & Hermes 源码分析
摘要 传统后端的容错逻辑很简单：try/except 捕获异常 → 重试 N 次 → 降级返回兜底值。这套范式在 Agent 系统中彻底失效了——不是因为机制不对，而是问题的性质变了。Agent 的容错不是"程序别崩"，而是"模型要知道发生了什么、有没有选择、下一步怎么走"。本文从 OpenClaw、Hermes、Claude Code 的工程实践中提炼出完整方案：三种失败分类 → 统一错误格式（情报层） → System Prompt 规则（决策层） → 循环检测（硬约束层） → 透明降级。核心洞察：把运维经验编码进工具返回结构里，把失败当作"观察结果"还给 LLM——这才是在 Agent 世界里真正有效的容错。

1. 两种完全不同的错误处理范式

1.1 传统后端：程序员写死一切

这是一个典型的传统容错代码：

// 传统后端：决策者是程序员，代码写死了运行时不会变
try {
    result = callPaymentAPI(request);
} catch (TimeoutException e) {
    if (retryCount < 3) {
        sleep(1000 * retryCount);   // 写死的退避策略
        retry();
    } else {
        return "支付失败";           // 写死的兜底行为
    }
}

决策者是程序员。所有分支都写死在代码里，运行时不会变。这在工作流系统中没有问题——一个支付接口的超时处理方式今天和明天一样。

1.2 Agent 模式：框架提供情报，模型自己做决策

但在 Agent 系统中，工具失败的原因千变万化：搜索结果为空、网页 403、API 限流、Python 脚本语法错误、Shell 命令被拒绝——程序员不可能穷举所有失败场景对应的处理逻辑。这正是 LLM 擅长的事：面对模糊、不确定、需要综合上下文的决策。

正确的思路是把失败变成 "观察结果"（observation） 还给模型：

// Agent 模式：决策者是 LLM，框架只提供结构化的"情报"
// 工具返回的不是异常，而是一份"观察报告"
{
    "success": false,
    "error_type": "TIMEOUT",
    "is_retryable": true,
    "suggestion": "Query 太长可能导致超时，试试缩短到 3 个词以内"
}
// → LLM 读到了这份"观察"
// → LLM 自己判断："我缩短 query 再试一次"
// → 或者："已经重试 2 次，这个方向走不通，换工具"
范式差异的核心：传统后端是"写死逻辑处理异常"，Agent 是"把失败当作观察结果还给模型"。前者依赖程序员的预见性，后者依赖模型的上下文推理能力。ReAct 循环的本质就是这种观察→思考→行动的闭环。

1.3 一个价值百万的生产事故

搜索工具的 error 返回格式不统一：有时是 {"error": "timeout"}，有时是 {"status": "failed", "reason": "timeout"}。模型对第二种格式不识别为失败，以为是有效结果——于是继续处理 → 发现不对 → 重新搜索 → 再次"成功"……循环一直持续到上下文耗尽。

根因不是模型笨，是工具返回格式不一致——模型识别不了失败信号。这就是为什么统一格式如此重要。

2. 三层 Harness 架构：什么归框架管，什么归模型管

从 OpenClaw、Hermes、Claude Code 和 Deep Research Agent 的实践中，可以提炼出一个通用的三层架构：

┌──────────────────────────────────────────────────────────────────┐ │ 🧠 决策层 │ Agent（LLM） │ │ "我缩短 query 重试" / "换工具" / "放弃并报告缺口" │ │ 在约束之内，综合上下文自由决策。模型越强，决策空间越大。 │ ├──────────────────────────────────────────────────────────────────┤ │ 📋 情报层 │ 框架（Harness） │ │ {success, error_type, is_retryable, suggestion} │ │ 把错误翻译成结构化"观察报告"。告诉模型：发生了什么、能不能重试。 │ ├──────────────────────────────────────────────────────────────────┤ │ 🛡️ 硬约束层 │ 框架（Harness） │ │ max_iterations / 循环检测 / token 预算 / 危险命令拦截 │ │ 绝对不能突破的安全边界。不给 LLM 选择权。 │ └──────────────────────────────────────────────────────────────────┘
层级负责人职责具体例子
硬约束层框架（Harness）绝对不能突破的安全边界，不给 LLM 选择权最大迭代次数到上限就停、循环检测强制中断、Token 预算耗尽即终止、危险命令模式匹配直接拦截
情报层框架（Harness）把错误翻译成结构化"观察报告"，告诉模型发生了什么统一返回 {success, error_type, is_retryable, suggestion}
决策层Agent（LLM）在约束之内综合上下文自由决策"缩短 query 重试" / "换搜索工具" / "告知用户并给出替代方案"
核心原则：硬约束不可协商，情报格式是协议不是建议，模型越弱 Harness 越重。强模型（Claude Opus / GPT-4 级别）可以给更大的决策空间，弱模型需要更具体的 suggestion，甚至把某些重试策略降级回硬编码。

3. 失败分类：不是所有失败都一样

3.1 临时性失败（可重试）

搜索 API 超时、网络抖动、服务器 500。特征是 "现在不行，等会儿可能就行"。

  • Harness 做的事：标记 is_retryable: true，建议指数退避（1s → 2s → 4s，最多 3 次）
  • Agent 做的事：判断是否值得重试、缩短 query 还是换参数

3.2 确定性失败（不可重试）

网页 404、403 无权限、Python 语法错误。特征是 "重试 100 次也没用"。

  • Harness 做的事：标记 is_retryable: false，给出具体的 failure reason
  • Agent 做的事：认识到"这个方向走不通"，自己决定换工具还是记录缺口继续

3.3 循环失败（需强制干预）

模型反复调用同一工具，每次都失败但检测不到，或者每次都微调参数但结果一样。特征是 "模型卡住了，需要外部干预"。

  • Harness 做的事：循环检测器用滑动窗口 + 语义相似度拦截，强制换策略或直接终止
  • 这是硬约束层，不给 LLM 选择权
三类失败对应三套处理机制：临时性 = Agent 决策 + Harness 建议；确定性 = Agent 换方向；循环 = Harness 直接干预。把它们混在一起用同一套 try/except 处理是最常见的生产事故来源。

4. 统一错误格式：模型能读懂的"失败协议"

所有工具的失败返回，不管是搜索超时还是网页 403，统一用一种格式。这就是"情报层"的落地：

{
    "success": false,
    "error_type": "TIMEOUT",          // 枚举值，模型可识别
    "error_message": "Search API timeout after 10s",
    "is_retryable": true,             // 告诉模型值不值得重试
    "suggestion": "Try a shorter or more specific query"  // 给模型一个锚点
}
字段作用为什么模型需要它
success布尔值标识是否成功模型一眼判断成败，不需要从文本里猜
error_type枚举值（TIMEOUT / FORBIDDEN / NOT_FOUND / PARSE_ERROR / RATE_LIMIT…）模型能按类型分流处理，而不是面对一堆模糊文本——这和 OpenClaw 的 FailoverError 设计思路一致
is_retryable布尔值无需模型自己判断"这个错误重试有没有意义"——把运维经验编码进去
suggestion可选的人类语言建议给模型一个"下一步怎么做"的锚点，减少随机决策。模型越弱这个字段越重要
设计原则：把人类运维经验编码进工具的返回结构里。操作文档里写"timeout 就重试，403 就别重试"——为什么不能直接让模型读到？统一格式就是为了这个。"情报层"的质量 = Agent 决策质量的上限。

5. System Prompt 规则：写在 prompt 里，不是写在代码里

有一个反直觉但很重要的结论：不应该让模型完全自由发挥。因为模型面对失败时，左右都是合理选择——重试、换参数、跳过、报告不足。如果没有规则约束，行为不可预测，QA 根本没法测。

规则写在 system prompt 里，不是写在代码里。这是和传统后端最大的区别：规则是给 LLM 读的"指导方针"，不是给 JVM 执行的字节码。

## 工具调用失败处理规则

1. 如果 is_retryable=True，最多重试 2 次，重试前修改参数（比如缩短 query）
2. 如果 is_retryable=False，记录"信息缺口：{失败的搜索目标}"，继续执行其他步骤
3. 如果同一工具在同一任务里已经失败 3 次，停止调用该工具，
   在最终报告里注明"以下信息未能获取：{列表}"
4. 如果核心信息无法获取，主动停止任务，
   输出"由于{原因}，本次研究无法完成，建议{替代方案}"

这和 Claude Code 的 system prompt 设计一致。其中有一条关键规则："Do not re-attempt the exact same tool call. If the user denies a tool, think about why and adjust your approach."

注意核心词：adjust your approach——不是"停下来等"，不是"无限重试"，是主动调整策略。工具失败不是终点，是信息——模型应该从这个信息里学到什么，然后做出更好的下一步决策。

6. 循环检测：最容易被忽视的安全阀

循环失败是硬约束层的典型代表——这里不给 LLM 选择权，检测到就强制干预。模型进入循环后，如果没有外部拦截，会一直跑到 token 耗尽或账单报警。

6.1 检测机制

class LoopDetector:
    def __init__(self, window_size: int = 5, similarity_threshold: float = 0.9):
        self.recent_actions = []
        self.window_size = window_size
        self.threshold = similarity_threshold

    def check(self, action: dict) -> bool:
        # 返回 True = 检测到循环，需要干预
        normalized = self._normalize(action)

        # 严格匹配：最近 5 步里有完全相同的 action？
        if normalized in self.recent_actions[-self.window_size:]:
            return True

        # 宽松匹配：语义相似度 > 90% 也认为在循环
        # 因为有时候 query 改了几个字但语义一样，纯字符串匹配检测不到
        for past_action in self.recent_actions[-self.window_size:]:
            if self._similarity(normalized, past_action) > self.threshold:
                return True

        self.recent_actions.append(normalized)
        return False

    def _normalize(self, action: dict) -> str:
        # 只保留工具名 + 核心参数，去掉时间戳等易变字段
        return f"{action['tool']}:{action.get('query', action.get('url', ''))}"

6.2 两级干预策略

阶段触发条件动作
强制换策略检测到相似 action 在滑动窗口内重复在下一步的 prompt 里注入提醒："你已经重复调用了相似的工具有 N 次，请更换思路或直接给出已有信息"
直接终止强制换策略后仍然循环终止任务，输出已有的中间结果——这是硬约束层的"最终否决权"
实际效果：某 Deep Research 项目加入循环检测后，因循环导致的 token 超量使用事故从每天 3 次降到接近 0。每次超量使用的成本可能是正常任务的 10～20 倍。这就是硬约束层的价值——在 LLM 卡住时果断介入，而不是等上下文耗尽。

7. 完整处理链路：从工具报错到最终报告

把以上所有组件串起来，一条完整的失败处理链路是这样的：

  1. 工具执行 → 返回统一结构（情报层）
    不管是成功还是失败，所有工具都返回 {success, error_type, is_retryable, suggestion} 格式。格式不一致会让模型"蒙眼决策"。
  2. 循环检测 → 判断是否重复（硬约束层）
    每次调用前检查最近 5 步，重复则强制换策略，仍然循环则终止。这里不给 LLM 选择权。
  3. System Prompt 规则 → 模型决策（决策层）
    可重试 → 重试并调整参数 / 不可重试 → 记录缺口继续 / 累计失败 3 次 → 放弃该工具。模型综合上下文自主判断。
  4. 降级策略 → 生成透明报告
    在最终报告里明确标注：哪些信息拿到了、哪些没拿到、置信度分级、建议补充途径。"不完整但诚实"远好于"完整但可能错误"。

8. 降级策略：信息拿不到时，Agent 应该说什么？

有些信息无论怎么重试就是拿不到。这时候正确的做法不是"凭空生成答案"，也不是"完全停下来"，而是透明地告知缺失，用已有信息给出尽可能有用的回答。

错误的降级

  • 基于不完整信息生成"看起来完整"的报告
  • 用户不知道有信息缺口
  • 结论可能错误但看起来很自信

→ 用户差评：结论不靠谱

正确的降级

  • 在报告里明确标注信息完整性说明
  • 区分高/中/低置信度
  • 列出未获取的信息 + 建议补充途径

→ 用户好评：不完整但可信

实现层面的最终报告模板：

## 研究结论
{conclusions}

## 信息完整性说明
本次研究共 {total_steps} 步搜索，成功获取 {success_count} 条。

**以下信息未能获取：**
{format_failed_queries(failed_queries)}
建议通过人工渠道补充上述信息。

## 置信度说明
- 高置信度（≥3个信源交叉验证）：{high_items}
- 中置信度（1-2个信源）：{medium_items}
- 信息不足（未获取）：{missing_items}

这和 Claude Code 的设计哲学完全一致："Be transparent about your limitations and uncertainties."——虽然不 impressive，但这是诚实的。

9. Claude Code 的特殊场景：用户拒绝工具权限

"用户拒绝"是一种特殊的工具失败——不是 API 超时，而是人类干预。这恰恰是 Agent 模式比写死逻辑更优雅的场景：代码无法预判用户为什么拒绝，但 LLM 可以从上下文推断。

Claude Code 的 system prompt 中有一条规则：

"If the user denies a tool you call, do not re-attempt the exact same tool call. Instead, think about why the user has denied the tool call and adjust your approach. If you do not understand why the user has denied a tool call, use AskUserQuestion."

三个递进策略：

层级做法适用场景
1. 推断原因从上下文推断为什么被拒（路径不安全？涉及隐私？），换个方式达到目标模型能确定原因时
2. 调整策略不重试同一件事，换一个工具或换一条路径有替代方案时
3. 主动问用户调用 AskUserQuestion 澄清："你拒绝是因为 X 吗？我换个方式可以吗？"看不懂原因时

这展示的是 Agent 模式的核心优势：把失败变成对话的起点，而不是流程的终点。传统后端在未知异常面前只能返回 500 或降级——Agent 可以主动探讨原因，把"异常"转成"新的信息输入"。

10. 面试框架：怎�回答"工具调用容错"

这道题考察的是你到底有没有把 Agent 放到生产环境里真实跑过。答题分三层：

第一层：分类（20 秒）

"失败分三类：临时性（可重试，指数退避）、确定性（不可重试，需告知模型）、循环失败（需强制干预）。不能统一用 try/except 跳过。"

第二层：架构（60 秒）

"三层分工：硬约束层（循环检测、迭代上限、token 预算——框架说了算）、情报层（统一错误格式，把运维经验编码进返回结构）、决策层（LLM 在约束内自主判断重试/换工具/放弃）。system_prompt 写规则让行为可测试，最终报告强制包含信息完整性说明。"

第三层：数据（40 秒）

"上线后异常报错率从 12% 降到 2%；循环超量使用从每天 3 次降到接近 0；用户对'透明地告知缺失'的评价高于'假装完整的错误答案'。"

面试官怎么判断你是真做过还是背答案：
背答案的人："我加了重试机制。"
真做过的人："循环检测器用最近 5 步的 action 指纹做比对，还加了语义相似度阈值——因为有时候 query 改了一点但语义是一样的，纯字符串匹配检测不到。这个 bug 是真的踩过之后才加进去的。还有一次搜索工具的 error 格式不统一，模型对第二种格式直接不识别为失败，循环跑到 token 耗尽才发现。后来所有工具返回全统一了格式。"

11. 总结：五层容错体系

#层级归属做什么为什么缺它不行
1失败分类框架把失败分成临时性 / 确定性 / 循环三类不同失败处理方式完全不同，混在一起统一跳过是危险操作
2统一格式框架（情报层）所有工具返回 {success, error_type, is_retryable, suggestion}格式不一致 = 模型思维混乱 = 决策质量崩塌
3System Prompt 规则框架制定 + LLM 遵守在 prompt 里写入失败处理规则：最多重试几次、什么时候放弃没有规则约束，行为不可预测，QA 没法测
4循环检测框架（硬约束层）滑动窗口 + 语义相似度检测，两级干预（换策略→终止）LLM 不会自己检测循环，外部不干预就一直烧 token
5透明降级框架制定 + LLM 执行在最终报告里标注信息完整性、置信度分级、建议补充途径"不完整但诚实" > "完整但可能错误"
一句话总结：传统后端把容错写成 if/else，Agent 把容错写成结构化观察 + 模型决策。三层分工（硬约束、情报、决策）缺一不可，统一格式是情报层的基础，循环检测是硬约束层最后的保险。把失败当作观察还给 LLM——这才是 Agent 容错的正确范式。
