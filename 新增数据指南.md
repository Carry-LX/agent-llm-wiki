# 新增数据指南

这份说明回答一个问题：以后要给这个 Agent 知识网络新增数据，应该如何判断 OneNote 里哪些内容是新增的，如何按现有数据格式写入项目，以及如何由 Codex 调用大模型直接阅读图片，生成图片描述、证据标题和节点名称。

核心原则：

- 用 OneNote 的最后修改时间戳判断“候选新增/变更页面”。
- 用 `page_id`、`onenote_object_id`、`image_sha256` 判断“真正新增/变更对象”。
- 图片文本、图片描述、证据标题、节点名称必须由 Codex 使用大模型直接阅读图片后生成。
- 不准依赖 OneNote 自带 OCR，也不要把本地 OCR 当作主要来源。现有 `ocr_text` 字段只能作为历史字段保留，新增流程不得读取它作为内容来源。
- 建图时必须同时区分“来源关系”和“语义关系”：OneNote 页面归属只能说明证据来自哪里，不能替代主题、概念和跨页语义关联。

## 先判断新增类型

新增数据分三类：

1. **新增 OneNote 原始材料**
   - 例如在 OneNote 的 `我的笔记本 / Agent` 分区里新增页面、图片、链接、文字。
   - 这是推荐方式，因为 `raw/`、`data/`、`wiki_generated/`、`app/wiki_data.*` 都应该由同步/生成流程统一产出。

2. **新增人工整理笔记**
   - 例如你对某个主题写了自己的总结、面试话术、判断标准。
   - 应该写到 `wiki_manual/`，不要写进 `wiki_generated/`。

3. **临时补一条网页展示数据**
   - 例如你暂时没有生成脚本，只想让 `app/index.html` 立刻多显示一个节点。
   - 可以手工改 `app/wiki_data.json` 和 `app/wiki_data.js`，但这只是临时办法；下次重新生成会被覆盖。

## 当前项目的真实情况

当前目录更像一个“生成结果包”。项目里有：

- `raw/`：OneNote 同步出来的原始数据、图片，以及历史富化结果。
- `data/`：图谱、主题、检索语料、标签缓存等结构化数据。
- `wiki_generated/`：自动生成的主题 Markdown。
- `wiki_manual/`：人工长期笔记。
- `app/`：本地网页和网页数据。

但当前目录里没有 README 提到的 `tools/` 目录。因此，如果要从 OneNote 正规新增数据，需要先找回或放回生成脚本，再重新生成这些输出文件。

## 正式增量新增流程

新增数据不要靠手工判断“我好像加了什么”。应该按下面的增量流程做。

### 1. 记录上次成功同步水位

以上次成功写入项目的时间作为水位，推荐记录在 `raw/last_sync_manifest.json`：

```json
{
  "schema_version": 1,
  "sync_run_id": "20260518T065825Z",
  "notebook": "我的笔记本",
  "section": "Agent",
  "generated_at_utc": "2026-05-18T06:58:30.3476414Z"
}
```

下一次同步时，把 `generated_at_utc` 当成 `last_success_sync_time_utc`。为了避免 OneNote、本机和脚本之间存在秒级误差，查询候选页面时建议向前回看 3-5 分钟。

```text
candidate_page if page_modified_time >= last_success_sync_time_utc - 5 minutes
```

### 2. 用页面修改时间找候选页面

从 OneNote 读取当前 `我的笔记本 / Agent` 分区的页面列表，至少拿到：

- `page_id`
- `page_title`
- `page_created_time`
- `page_modified_time`
- `page_index`

判断逻辑：

- `page_id` 不在旧的 `raw/onenote_pages.jsonl` 中：新页面。
- `page_id` 已存在，但 `page_modified_time` 晚于旧记录：变更页面。
- `page_modified_time` 没变：跳过该页。

注意：OneNote 的修改时间只能告诉我们“这一页可能变了”，不能直接告诉我们是哪张图片或哪段文字变了。因此对候选页面要重新解析整页。

### 3. 对候选页面重新解析对象

对每个候选页面，重新解析页面 XML 或页面内容，提取：

- 页面文字块。
- 图片对象。
- 图片附近的链接。
- 图片在页面里的顺序。
- 对象创建/修改时间。

图片对象至少需要这些字段：

```json
{
  "schema_version": 1,
  "asset_id": "agent_img_002_035_xxxxxxxxxxxx",
  "asset_type": "onenote_image",
  "sync_run_id": "20260519T010203Z",
  "notebook": "我的笔记本",
  "section": "Agent",
  "page_title": "RAG",
  "page_index": 2,
  "page_id": "...",
  "page_modified_time": "2026-05-19T01:00:00.000Z",
  "image_index": 35,
  "onenote_object_id": "...",
  "object_created_time": "2026-05-19T00:58:00.000Z",
  "object_modified_time": "2026-05-19T00:59:00.000Z",
  "source_url": "https://example.com/article",
  "image_path": "raw/images/agent_img_002_035_xxxxxxxxxxxx.png",
  "image_sha256": "...",
  "image_bytes": 123456,
  "width": "800",
  "height": "600"
}
```

### 4. 判断图片是新增、变更还是旧图

先加载旧的 `raw/assets.jsonl` 或 `raw/assets_enriched.jsonl`，建立三个索引：

- `page_id + onenote_object_id`
- `image_sha256`
- `asset_id`

判断规则：

- `onenote_object_id` 不存在，且 `image_sha256` 不存在：新增图片。
- `onenote_object_id` 已存在，但 `image_sha256` 变化：同一 OneNote 对象的图片内容变更。
- `image_sha256` 已存在，但出现在新页面或新位置：重复图片引用，需要合并或记录重复位置。
- `onenote_object_id` 和 `image_sha256` 都没变：旧图片，跳过图像理解，沿用旧的 enriched 字段。

`image_sha256` 是最终去重依据。网页里的“原始证据”和“去重证据节点”就是因为同一图片可能在 OneNote 多处出现。

### 5. 用 Codex 大模型直接读图

当你要求 Codex 帮你新增数据时，新增图片必须由 Codex 调用具备视觉理解能力的大模型直接阅读图片。这里的“大模型读图”是内容来源，不能偷懒换成 OneNote OCR，也不应把本地 OCR 当作主要来源。

- 不读取 OneNote 导出的 OCR 文本作为 `best_text`。
- 不用 OneNote OCR 生成 `clean_text`。
- 不用 OneNote OCR 生成节点名称。
- 不用 OneNote OCR 做主题分类。
- 不以 Tesseract、PaddleOCR、RapidOCR 等本地 OCR 作为主要理解来源。
- 如果图片里有复杂表格、流程图、截图界面，应由大模型综合视觉布局、文字、箭头、层级关系生成说明。

如果旧数据里已经有 `ocr_text`，可以保留字段用于兼容，但新增逻辑必须把它当作“不可信历史字段”。

Codex 读图后，输出至少包括：

```json
{
  "vision_text": "大模型从图片中读出的关键文字，保留层级和结构。",
  "vision_description": "大模型对图片内容、结构和含义的自然语言说明。",
  "clean_text": "基于大模型读图结果整理出的可检索文本。",
  "best_text": "最终用于检索、图谱和网页详情的文本。",
  "best_text_source": "codex_vision_model",
  "label_title": "适合作为证据节点名的短标题",
  "label_summary": "一句话概括这张图支撑什么观点",
  "label_keywords": ["RAG", "评测集", "多跳推理"],
  "label_model": "Codex 使用的视觉大模型名",
  "label_generated_at": "2026-05-19T01:02:03Z"
}
```

字段含义：

- `vision_text`：图片里的文字内容，不要求逐字机械转录，但要保留关键术语、数字、层级和关系。
- `vision_description`：解释这张图在讲什么，尤其是流程、表格对比、架构图、截图界面的含义。
- `clean_text`：适合搜索和后续生成的整理文本。
- `best_text`：`vision_text + vision_description` 融合后的最终证据文本。
- `label_title`：图谱节点名。

如果确实需要辅助校验，可以让本地 OCR 做旁路参考，但不能让它覆盖大模型读图结论；最终写入字段应以 Codex 大模型视觉理解为准。

### 6. 生成证据节点名称

证据节点名称不要直接用图片文件名，也不要直接用图片第一行文字。应该由 Codex 大模型理解图片内容后生成一个短标题。

标题要求：

- 8-18 个中文字符左右，必要时可含英文术语。
- 能表达图片核心内容。
- 不要写成“截图”“图片”“表格”这种泛称。
- 避免和已有节点重名；重名时加关键限定词。

示例：

```text
RAG评测集构建六步
向量数据库必要性
GraphRAG与PathRAG对比
SFT观察区Loss Mask
Text2SQL执行前校验
```

节点里建议写入：

```json
{
  "id": "asset_imgsha_cd654a4171e8",
  "type": "evidence",
  "title": "RAG评测集构建六步",
  "description": "图片说明 RAG 测试集从评估范围、问题设计、参考答案、支持证据到质量审查的构建流程。",
  "text": "清洗后的图片文本和模型理解摘要",
  "image_path": "raw/images/agent_img_001_002_cd654a4171e8.png",
  "source_url": "https://mp.weixin.qq.com/s/GUeJCJfd03cSK5aCoefRaA",
  "page_title": "Agent",
  "asset_id_raw": "agent_img_001_002_cd654a4171e8",
  "label_model": "codex_vision_model",
  "label_generated_at": "2026-05-19T01:02:03Z"
}
```

### 7. 主题归类和图谱关系

新增证据要归到现有主题，除非真的需要新增主题。

现有主题来自 `data/topics.json`：

- `rag_and_retrieval`
- `agent_architecture`
- `planning_execution`
- `prompt_engineering`
- `fault_tolerance_permission`
- `text2sql`
- `finetuning_trajectories`
- `evaluation_dataset`

归类时使用 `best_text`、`vision_description`、`label_keywords` 和页面标题。不要使用 OneNote OCR，也不要只靠本地 OCR。

每个新增证据至少要有一条边：

```json
{
  "source": "topic_rag_and_retrieval",
  "target": "asset_imgsha_cd654a4171e8",
  "relation": "supported_by",
  "relation_label": "由证据支撑",
  "weight": 8
}
```

如果能识别到概念，还可以补：

```json
{
  "source": "concept_eval_dataset",
  "target": "asset_imgsha_cd654a4171e8",
  "relation": "explained_by",
  "relation_label": "由材料说明",
  "weight": 8
}
```

#### 避免只按 OneNote 页面归属建图

后续新增或重构生成脚本时，一定要避免把图谱做成“页面目录图”。每个页面、主题、概念和证据之间至少要分清下面几类关系：

1. **来源关系**
   - 表示证据原始来自哪个 OneNote 页面。
   - 典型关系：`contains_evidence` / `页面包含证据`。
   - 这类关系只能回答“材料来自哪里”，不能回答“材料属于哪个知识点”。

2. **主题关系**
   - 表示证据或页面属于哪个知识主题。
   - 典型关系：`supported_by` / `由证据支撑`，或 `page_topic` / `关联主题`。
   - 例如 `RAGAS评估与Doc2Query` 页面既应连到 `评测集与数据构造`，也可以连到 `RAG 与检索增强`。

3. **概念关系**
   - 表示页面或证据解释了某个抽象概念、方法或机制。
   - 典型关系：`explained_by` / `由材料说明`，或 `page_concept` / `关联概念`。
   - 例如 `记忆` 页面应连到 `Agent 记忆机制`，`Text2SQL` 页面应连到 `Text2SQL 双重校验`。

4. **跨页语义关系**
   - 表示证据虽然来自别的 OneNote 页面，但内容上属于当前页面或概念。
   - 典型关系：`semantic_evidence` / `语义相关证据`。
   - 例如 Memory 相关证据可能来自 `Claude code`、`RAG` 或 `杂项` 页面，但点击 `记忆` 页面时仍应直接看到这些证据。

不要只做这一种边：

```json
{
  "source": "root_agent_wiki",
  "target": "page_003",
  "relation": "contains_page",
  "relation_label": "包含页面",
  "weight": 1
}
```

这种边只能说明页面在分区下面，不能说明这个页面和哪些证据、主题、概念相关。对于标题很短的页面尤其要额外检查，例如：

- `记忆`
- `提示词`
- `Planning`
- `容错 / 权限`
- `Text2SQL`

这些页面标题通常是主题名，而不是具体材料名。生成图谱时要用 `best_text`、`vision_description`、`label_keywords`、页面标题和已有主题/概念节点一起判断，不要只看“证据原来放在哪个 OneNote 页面里”。

重构生成脚本时建议加入一个固定检查：

```text
for each page node:
    direct_neighbors = graph.neighbors(page)
    if direct_neighbors only contains root or contains_page edges:
        mark as suspicious

    semantic_candidates = evidence whose title/text/keywords match page title, topic, concept, or aliases
    if semantic_candidates exist but page has no direct semantic_evidence/page_topic/page_concept edges:
        add or review semantic edges
```

低连接页面不一定错，但要人工确认。一般来说，一个分区页如果已经有明显主题含义，却只连接中心节点或只连接 1 条本页证据，就需要检查是否漏了跨页语义关系。

### 8. 增量写回哪些文件

新增或变更后，需要更新这些文件：

- `raw/onenote_pages.jsonl`：更新候选页面记录。
- `raw/assets.jsonl`：追加或更新图片原始资产记录，但不要把 OneNote OCR 当作有效内容。
- `raw/assets_enriched.jsonl`：写入 Codex 大模型读图结果。
- `raw/images/`：保存新增图片。
- `data/search_corpus.jsonl`：写入 `best_text` 和元信息。
- `data/graph_nodes.jsonl`：写入新增/更新节点。
- `data/graph_edges.jsonl`：写入新增/更新关系。
- `data/graph.json`：重建总图。
- `data/topics.json`：主题证据数变化时更新。
- `wiki_generated/*.md`：自动主题页重建。
- `app/wiki_data.json`：网页 JSON 数据重建。
- `app/wiki_data.js`：网页 JS 数据重建。

建议不要在多个文件里分别手搓同一条数据。更稳的做法是以 `raw/assets_enriched.jsonl` 为可信源，重新生成 `data/`、`wiki_generated/` 和 `app/wiki_data.*`。

### 9. 增量同步伪代码

```text
last_sync_time = read raw/last_sync_manifest.json.generated_at_utc
candidate_pages = list_onenote_pages_modified_after(last_sync_time - 5min)

old_pages = load raw/onenote_pages.jsonl
old_assets = load raw/assets_enriched.jsonl
old_by_object = index old_assets by page_id + onenote_object_id
old_by_sha = index old_assets by image_sha256

for page in candidate_pages:
    page_xml = fetch_page_xml(page.page_id)
    parsed_page = parse_page(page_xml)

    upsert raw/onenote_pages.jsonl with parsed_page metadata

    for image in parsed_page.images:
        save image to raw/images/
        image_sha256 = sha256(image bytes)

        if old_by_object has same object and same sha:
            reuse old enriched fields
            continue

        if old_by_sha has same sha:
            merge duplicate location
            continue

        vision_result = codex_read_image_with_vision_model(image)
        label_result = generate_title_summary_keywords(vision_result)
        enriched_asset = merge raw image metadata + vision_result + label_result
        write enriched_asset

rebuild graph/search/wiki/app data from enriched assets
write new raw/last_sync_manifest.json only after all outputs succeed
```

### 10. 失败处理

只有当以下文件都成功写完，才能更新 `raw/last_sync_manifest.json` 的 `generated_at_utc`：

- `raw/assets_enriched.jsonl`
- `data/graph.json`
- `app/wiki_data.json`
- `app/wiki_data.js`

如果中途失败，不要推进水位。否则下一次会误以为这些 OneNote 修改已经同步过，从而漏数据。

## 推荐流程：从 OneNote 新增

1. 在 OneNote 的 `我的笔记本 / Agent` 分区新增材料。
   - 新页面会成为新的 page 节点。
   - 页面里的图片会成为 evidence 证据。
   - 图片附近的链接或页面文本会进入来源和检索语料。

2. 运行同步/生成流程。
   - 如果项目恢复了 `tools/` 目录，按 README 的说明运行。
   - 如果 `tools/` 仍不存在，不要直接乱改 `raw/` 和 `data/`，应先找到原来的生成脚本来源。

3. 检查这些输出是否更新：
   - `raw/onenote_pages.jsonl`
   - `raw/assets.jsonl`
   - `raw/assets_enriched.jsonl`
   - `raw/images/`
   - `data/topics.json`
   - `data/graph.json`
   - `data/graph_nodes.jsonl`
   - `data/graph_edges.jsonl`
   - `data/search_corpus.jsonl`
   - `wiki_generated/*.md`
   - `app/wiki_data.json`
   - `app/wiki_data.js`

4. 打开 `app/index.html` 检查网页。
   - 左侧统计里的“原始证据”和“去重证据节点”应该变化。
   - 新主题或新证据应该能在搜索框里搜到。
   - 右侧详情里应该能看到原图、来源链接、整理文本。

## 人工笔记应该放哪里

如果你只是想新增自己的理解，不要改自动生成文件。

推荐放在：

- `wiki_manual/Agent分区初版总览.md`
- 或者在 `wiki_manual/` 下新建一个主题文件，例如 `wiki_manual/RAG人工补充.md`

适合写进 `wiki_manual/` 的内容：

- 面试回答模板。
- 自己总结的技术判断。
- 多篇材料融合后的观点。
- 对某个主题的长期修订。
- 不希望下次自动生成时被覆盖的内容。

不要写进：

- `wiki_generated/`：会被自动生成覆盖。
- `data/`：这是结构化中间产物。
- `app/wiki_data.*`：这是网页展示数据。

## 临时手工新增网页数据

只有在没有生成脚本、并且只是临时展示时，才建议这么做。

最小要改两个文件：

- `app/wiki_data.json`
- `app/wiki_data.js`

这两个文件内容要保持一致。`wiki_data.js` 的格式是：

```js
const WIKI_DATA = {
  ...
};
```

### 手工新增一个证据节点的大致步骤

1. 把图片放到 `raw/images/`。

2. 在 `app/wiki_data.json` 的 `graph.nodes` 里新增一个节点，至少需要：

```json
{
  "id": "asset_manual_xxx",
  "type": "evidence",
  "title": "这里写证据短标题",
  "description": "这里写整理后的说明",
  "image_path": "raw/images/你的图片.png",
  "page_title": "对应 OneNote 页面",
  "source_url": "可选来源链接"
}
```

3. 在 `graph.edges` 里加关系，把它连到主题、页面或概念上：

```json
{
  "source": "topic_rag_and_retrieval",
  "target": "asset_manual_xxx",
  "relation": "supported_by",
  "relation_label": "由证据支撑",
  "weight": 3
}
```

4. 如果希望搜索语料也包含它，还要同步更新 `data/search_corpus.jsonl`。

5. 把更新后的 `app/wiki_data.json` 内容同步到 `app/wiki_data.js`，外层加上 `const WIKI_DATA = ...;`。

### 手工新增主题时还要注意

如果新增的是主题节点，除了 `graph.nodes` 和 `graph.edges`，还要同步更新：

- `data/topics.json`
- `wiki_generated/index.md`
- 可能还要新增 `wiki_generated/主题名.md`
- `app/wiki_data.json` 顶层的 `topics`

主题节点 id 建议保持这种格式：

```text
topic_英文或拼音标识
```

例如：

```text
topic_agent_memory
```

## 文件职责表

| 文件或目录 | 是否推荐手工改 | 用途 |
|---|---:|---|
| `raw/images/` | 可以追加 | 存图片原件 |
| `raw/assets.jsonl` | 不推荐 | OneNote 图片资产原始记录 |
| `raw/assets_enriched.jsonl` | 不推荐 | Codex 大模型读图后的证据文本、描述和节点标签 |
| `raw/onenote_pages.jsonl` | 不推荐 | OneNote 页面元信息 |
| `data/topics.json` | 临时可改 | 主题列表和主题统计 |
| `data/graph.json` | 临时可改 | 图谱总数据 |
| `data/graph_nodes.jsonl` | 临时可改 | 图谱节点 |
| `data/graph_edges.jsonl` | 临时可改 | 图谱关系 |
| `data/search_corpus.jsonl` | 临时可改 | 搜索语料 |
| `wiki_generated/` | 不推荐 | 自动生成的主题文档 |
| `wiki_manual/` | 推荐 | 人工长期整理 |
| `app/wiki_data.json` | 临时可改 | 网页读取的数据 |
| `app/wiki_data.js` | 临时可改 | 网页直接加载的数据 |
| `app/index.html` | 可以改 | 网页界面和交互 |

## 新增后的检查清单

新增数据后至少检查这些点：

- `app/index.html` 能打开。
- 左侧统计数量符合预期。
- 搜索框能搜到新增节点。
- 点击新增节点后，右侧详情能显示标题、页面、来源、图片或整理文本。
- 新节点至少有一条边连接到主题、页面或根节点，否则图谱里会变成孤点。
- 页面节点不能只依赖 `root_agent_wiki -> page` 这种中心包含关系；如果页面本身是主题页，还应至少连到对应 `topic_*` 或 `concept_*` 节点。
- 对短标题主题页要检查跨页语义证据，例如 `记忆`、`提示词`、`Planning`、`容错 / 权限`、`Text2SQL`。如果相关证据分散在其他 OneNote 页面，需要补 `semantic_evidence`。
- 点击每个新增或变更页面节点时，右侧“相邻关系”应能看到实际相关的证据、主题或概念，而不是只有中心节点。
- 如果新增图片，`image_path` 必须能从 `app/index.html` 位置访问，通常写成 `raw/images/xxx.png`，页面代码会自动加 `../`。
- JSON 里不要出现重复 id。
- 文件保存为 UTF-8，换行使用 LF。

## 建议的长期做法

长期最好恢复完整生成链路，把新增流程固定成：

```text
OneNote 新增材料
  -> 同步 raw/
  -> Codex 大模型读图富化 raw/assets_enriched.jsonl
  -> 重建 data/
  -> 重建 wiki_generated/
  -> 重建 app/wiki_data.json 和 app/wiki_data.js
  -> 打开 app/index.html 检查
```

这样 `app/` 只是展示层，`data/` 是结构化中间层，`wiki_manual/` 保存人的判断，整个项目不会因为手工补数据而越来越乱。
