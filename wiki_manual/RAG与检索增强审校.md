## 检索技术

### HNSW 分层贪心搜索

HNSW 检索的正确理解是分层贪心定位而非每层保留 Top-N 展开：
- 高层每层只找 1 个当前最优入口节点，仅将其带到下一层。
- 底层（第 0 层）才用候选池 ef 扩大搜索范围，输出最终 Top-K 结果。

常见错误是认为高层每层保留 Top-N 再分别向下搜索。

对应证据：`agent_img_002_025_f5c936ace146`、`agent_img_002_022_14f94fda7909`。

### IVF 聚类桶缩小搜索范围

IVF 先对向量做聚类，把相似向量分进同一个桶，查询时只搜最相关几个桶。适合对内存和延迟敏感的大规模场景，但聚类质量直接影响召回率。

对应证据：`agent_img_002_023_94234dd7d215`。

### 向量数据库的存在价值

专门的向量数据库（如 Milvus）存在的理由是 MySQL 和 ES 对高维向量的相似度搜索没有原生索引支持，ANN 索引（HNSW、IVF 等）是向量数据库的核心加速结构。

对应证据：`agent_img_002_021_9b3378ea7e27`。

### Metadata 先过滤再 ANN

实际业务中应先按 Metadata（部门、时间范围等）过滤范围，再在过滤结果上执行 ANN 搜索。这比全库 ANN 后再过滤更高效，也避免了跨租户信息泄露。

对应证据：`agent_img_002_028_b64d5058598d`。

### SPLADE 稀疏与语义融合

混合检索的前沿方向是 SPLADE 这类稀疏向量模型，它把 BM25 的精确匹配能力和向量的语义泛化能力结合到一个模型中，实现更深层次的"稀疏+语义"融合。

对应证据：`agent_img_002_016_99f8be553c37`。

### 代码检索中 grep 优于向量搜索

代码搜索场景下，函数名、类名、常量是高精度锚点，精确匹配（grep）恰好是最直接的检索方式。不要在所有场景都默认上向量检索。

对应证据：`agent_img_010_011_2fe9a3cb897f`。

## Embedding

### Embedding 模型演进

Embedding 技术经历了三代演进：
1. 静态词向量（Word2Vec、GloVe）：固定映射，无法处理多义词。
2. 上下文感知（BERT 系）：同一个词在不同上下文中向量不同。
3. 指令微调（Instructor 系）：模型能理解任务指令，通用性更强。

对应证据：`agent_img_002_031_6f0b11003f01`。

### 中文 RAG 优先 BGE

常见 Embedding 模型选择：OpenAI text-embedding-3-small 性价比高、英文效果好但有费用和数据出境问题；BGE 系列中文效果好、开源可本地部署。中文 RAG 场景优先考虑 BGE-large-zh。

对应证据：`agent_img_002_030_7d149ed57d08`。

### Embedding 模型评估要用业务数据

评估 Embedding 模型不能只看 MTEB 等通用排行榜分数。MTEB 高分不代表在特定业务数据上表现好，实际选型应在业务数据集上做召回率对比。

对应证据：`agent_img_002_029_b54e7ba8ca9a`。

## 文档切分

### 层级划分 + 句子级 Overlap

最佳实践是文档层级划分结合句子级 overlap，而非固定长度截断：
- 层级划分保留原文档的标题和段落结构信息。
- 句子级 overlap 基于句子边界调整，避免在句子中间切断导致语义碎片化。

对应证据：`agent_link_001_002_3aa486062719`、`agent_link_002_001_43ab2992d0b1`、`agent_img_002_013_046fc4a9b9b8`。

## RAG 质量治理

### 五层质量治理框架

RAG 质量治理应覆盖五层：
1. **幻觉来源三层**：解析层（文档质量）→ 检索层（召回精度）→ 生成层（模型约束）。
2. **治理优先级**：Prompt 约束最快 → 检索优化最重要 → 置信度兜底最稳。
3. **拒答机制**：相关性阈值 + 引用覆盖率双重判断，配合话术设计保用户体验。
4. **更新同步**：增量索引 + 版本管理 + 缓存分层失效，三件事缺一不可。
5. **评测闭环**：离线黄金集（覆盖三类题）+ 在线用户信号 + 每周回捞迭代。

对应证据：`agent_img_002_017_644feb67b2da`。

### Badcase 闭环

RAG 系统上线后应建立完整闭环：badcase 收集 → 分类（检索失败/幻觉/路由错误/知识缺失）→ 按类分配修复 → 原 case 回归验证 → 全量回归 → 灰度发布。不应临时修改 prompt 或单条规则了事。

对应证据：`agent_link_002_003_94fb83bf7cee`。

### 动态 RAG 数据质量评估

不能只看单次检索效果，需从文档质量、检索召回、生成准确性和用户反馈四个维度建立持续评估体系，并在系统迭代中追踪各维度指标变化。

对应证据：`agent_link_002_005_76cb9b23e50d`。

## RAG 幻觉

### 幻觉来源在生成阶段

RAG 最致命的幻觉问题并非检索失败——明明检索到了正确文档片段，模型生成时却出现了文档中不存在的信息。这是 LLM 在生成阶段将参数知识与检索内容混合的结果。

对应证据：`agent_img_001_007_d75dc63bc8b4`、`agent_img_002_008_d75dc63bc8b4`。

### RAG Prompt 的根本矛盾

LLM 预训练时已存储大量参数知识。给定检索文档后，模型的默认行为是将文档信息与自身参数知识混合使用。因此 RAG Prompt 必须显式约束模型只能使用传入文档片段，而不能依赖参数知识。

对应证据：`agent_img_001_018_90388fa1fee2`。

## 权限控制

RAG 系统在金融等场景需要三层权限控制：
1. **向量数据库过滤层**：通过元数据过滤（expr 条件）限制检索范围。
2. **分区物理隔离层**：不同租户数据分区分库。
3. **返回层二次验证**：对检索结果做逐条权限校验，确保无越权数据泄露。

对应证据：`agent_img_001_011_6947aa966103`、`agent_img_002_012_6947aa966103`。

## 性能优化

### 三层缓存体系

1. **Embedding 缓存**：避免重复计算向量，embedding 是最"浪费钱"的环节。
2. **检索缓存**：热点查询结果缓存，减少重复检索。
3. **Answer 缓存**：完全相同的问答对直接返回，跳过全链路。

对应证据：`agent_img_009_016_93a7879f1211`、`agent_img_009_017_456194efbeca`。

### RAG 延迟优化

RAG 全链路四步：Embedding → 向量检索 → Prompt 拼装 → LLM 生成。首字延迟（TTFT）主要卡在 Embedding 和向量检索。优化方向包括异步并行、预计算 embedding、检索结果缓存。

对应证据：`agent_img_009_018_674942d618e0`、`agent_img_009_015_8caf164bbed7`。

### 向量索引内存与写入

- 向量索引占满内存触发 swap 时，查询延迟可能从毫秒级飙升到秒级。工程上应优先检查内存和量化策略。
- 批量写入会触发 Segment 合并造成查询抖动，应错峰到低峰期，拆成 500-1000 条小批次降低 P99 延迟冲击。

对应证据：`agent_img_002_027_e76d357cd694`、`agent_img_002_026_f9760a189147`、`agent_img_002_024_e00ff7ab6236`。

## 检索增强技术

### HyDE 用假设答案增强检索

HyDE 的思路是让 LLM 先根据 query 生成一个假设的理想回答，再用这个回答的 embedding 去检索（而不是直接用 query 的 embedding）。这样可以弥补短 query 和长文档之间的语义鸿沟，但有额外推理成本。

对应证据：`agent_img_001_009_94911bd17c4e`、`agent_img_002_010_94911bd17c4e`。

### Reranker 后必须设相关性阈值

Reranker 给每条文档打了 0-1 的相关性分数后按分数取 Top-K，但如果没有设置最低阈值，可能出现 Top-K 全部不相关的情况。应同时设 Top-K 上限和最低相关性阈值。

对应证据：`agent_img_002_014_997cdfa9a54b`。

### GraphRAG 对比

| 方案 | 检索机制 | 特点 |
|---|---|---|
| GraphRAG | 社区检测找到相关社区并聚合内容 | 适合宏观摘要类问题 |
| LightRAG | 双阶段：快速局部检索 + 深层全局检索 | 兼顾效率和深度 |
| PathRAG | 基于流的剪枝算法识别检索路径 | 适合推理链问题 |

对应证据：`agent_img_002_001_dcd9267f7f8b`。

### 查询路由

不同查询应路由到不同链路：事实型查询走知识库检索，分析型查询走推理链路，闲聊走纯对话。不能所有 query 都走同一条路。

对应证据：`agent_img_001_012_922c16272133`。
