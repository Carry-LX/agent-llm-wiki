# LLM 推理机制：Prefill 与 Decode

可以把 LLM 生成回答理解成两步：**先读题，再一个字一个字答题**。

## 1. Prefill 阶段：把你的问题"读完并理解"

比如你输入：

> "请解释一下 LLM 的 Prefill 和 Decode 阶段是什么意思。"

模型不会一边读第一个字一边立刻回答，而是先把你整段 prompt 一次性处理完。

这个阶段做的事大概是：

> 把用户输入的所有 token 都送进模型，计算它们之间的关系，并生成后面回答时要用的中间结果，也就是 **KV Cache**。

你可以理解成：

> **Prefill = 模型读完题目、做完笔记。**

比如 prompt 有 1000 个 token，Prefill 阶段会把这 1000 个 token 一起处理掉。因为这一步可以大规模并行计算，所以更依赖 GPU 的计算能力，也就是"计算密集型"。

---

## 2. Decode 阶段：开始"一个 token 一个 token 地生成答案"

Prefill 结束后，模型开始正式输出回答。

比如它要生成：

> "Prefill 阶段可以理解为……"

它不是一次性把整句话吐出来，而是这样生成的：

- 第 1 步：生成 "Prefill"
- 第 2 步：根据前面的内容，再生成 "阶段"
- 第 3 步：再生成 "可以"
- 第 4 步：再生成 "理解"
- ……

也就是说：

> **Decode = 模型根据已经读完的 prompt 和已经生成的内容，一个 token 一个 token 地往外写答案。**

这个过程不能完全并行，因为下一个 token 依赖上一个 token。

---

## 3. KV Cache 是什么？

KV Cache 可以先简单理解成：

> 模型在读 prompt 和生成回答时留下的"记忆笔记"。

模型每生成一个新 token，都需要参考之前的上下文。如果每次都重新计算整个上下文，会非常慢。

所以模型会把之前计算好的 Key/Value 信息缓存起来，这就是 KV Cache。

它的作用是：

> 避免重复计算，让 Decode 阶段可以直接复用之前的结果。

但问题是，Decode 阶段每生成一个 token，都要频繁读取这些 KV Cache，所以它特别依赖显存带宽和内存访问速度。

---

## 4. 为什么说两个阶段资源需求相反？

### Prefill：像一次性读一大段文章

它要处理整个 prompt，计算量大，但可以并行。

特点是：

> **GPU 算力用得多。**

所以说它是 **计算密集型**。

### Decode：像一个字一个字写答案

它每次只生成一个 token，但是每次都要查之前的 KV Cache。

特点是：

> **大量读写显存，计算量反而没那么大。**

所以说它是 **内存密集型**。

---

## 5. 用一个简单类比

你可以把模型想象成学生考试：

**Prefill 阶段：**

> 学生先把题目和材料全部读完，划重点，做笔记。

这个时候主要是在"理解题目"，工作量大，但可以集中处理。

**Decode 阶段：**

> 学生开始写答案，每写一句都要回头看笔记和前文，确保下一句合理。

这个时候不是一次性写完整篇，而是一点点往下写。

---

## 6. 为什么混在一起跑会互相干扰？

因为 Prefill 和 Decode 对 GPU 的需求不一样。

Prefill 想要的是：

> "给我大量算力，我要一次性算很多东西。"

Decode 想要的是：

> "给我稳定、低延迟的显存访问，我要不断读取 KV Cache。"

如果很多请求同时跑，有的在 Prefill，有的在 Decode，就会出现资源抢占。

比如：Prefill 请求突然进来，占用了大量 GPU 计算资源；Decode 请求本来需要稳定地一个 token 一个 token 输出，结果被 Prefill 干扰，生成速度变慢，用户就感觉回答卡顿。

所以现在很多推理系统会尝试把 Prefill 和 Decode 拆开调度，甚至放到不同类型的机器上跑。

## 一句话总结

> **Prefill 是模型"读完问题并建立上下文"的阶段，主要吃算力；Decode 是模型"逐 token 写答案"的阶段，主要吃显存带宽和 KV Cache 访问。**
