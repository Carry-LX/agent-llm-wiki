## 微调评估

### 工具调用评估四维度

Agent 工具调用微调不能只看总体准确率，必须拆分四个维度评估：

1. **工具选择准确率**：是否选了正确的工具。
2. **参数格式准确率**：工具名选对了，但参数格式写错的比例。
3. **调用顺序准确率**：多工具任务中，顺序是否合理（不该先计算就先搜索）。
4. **遗忘率**：微调后在非工具调用任务（普通问答）上的表现有没有明显下降。

最后一项尤其重要。SFT 存在遗忘风险：专项微调做得太猛，模型在专项任务上提升但在其他任务上变差。

### 防遗忘策略

训练数据里混入约 30% 通用指令数据——不能全是工具调用样本，否则模型会忘记怎么做普通问答。这个比例是实验得出的，不是拍脑袋定的。

实践中 FC-SFT 5000 样本可将工具调用准确率从 71% 提升到 94%，同时 MMLU 下降仅 0.3% 在统计误差范围内。

## 训练数据构造

### 人工标注 + 模型蒸馏组合

业界常用组合策略：

| 方法 | 规模 | 质量 | 用途 |
|---|---|---|---|
| 人工标注员逐条手写 | 几千条 | 最高 | 核心种子数据，质量基准 |
| GPT-4 等强模型批量生成 | 几十万条 | 中等 | 规模化训练 |
| 人工抽查过滤 | — | — | 去重、去噪、质检 |

核心风险是**幻觉传递**：上游模型错一个，下游学一个。因此少量人工保质量 + 大量蒸馏保规模的组合不可或缺。

对应证据：`agent_img_005_006_82adbd19533c`。

## Agent 自进化

### Hermes 双路径自进化闭环

Hermes Agent 的自进化依赖两条互补路径：

| 维度 | 路径一：自动 Skill 生成 | 路径二：RL 训练 |
|---|---|---|
| 触发方式 | 运行时每 N 轮对话自动触发 | rl_cli.py 手动发起 |
| 生效速度 | 即时（下一轮对话就能用） | 需要数小时训练 |
| 改变什么 | Agent 的上下文知识（Skill Memo），不改模型 | 模型本身的权重（内在能力） |
| 类比 | 记笔记 | 练内功 |
| 优势 | 快速、轻量、可解释、允许人工干预纠偏 | 根本上改变模型能力，突破天花板 |
| 局限 | 模型权重始终没变，天花板明显 | 训练成本高，可解释性差 |

两条路径构成"内外"双轮驱动的自进化闭环，覆盖从快速响应到深层能力提升的全频谱。

对应证据：`agent_img_009_004_7a49da56130f`。

### 上下文工程的天花板

基于 Skill 沉淀的上下文工程进化存在根本性局限：无论积累多少 Skill，底层模型权重始终没变。Agent 只是在不断检索外部知识库，而非将经验内化为自身直觉和能力。

对于追求极致性能或特定垂直领域需要突破通用模型瓶颈的场景，基于 Context Engineering 的优化方式存在天花板。必须引入 RL 训练闭环，通过改变模型权重实现真正的能力自进化。

对应证据：`agent_img_009_007_6d1f81043b69`。
