# Prompt 约束与上下文设计审校

这页整理 Prompt 约束与上下文设计主题下的关键结论。目标是把自动页里受 OCR 影响的零散摘录，整理成可复用的工程判断。

- 2 条因内容实质不属于 Prompt 主题，重归类到其他主题。

## 结构化输出

### 技术路线全景

大模型结构化输出可按技术路线分为六类，每类有明确的适用边界：

| 路线 | 机制 | 核心优势 | 主要局限 |
|---|---|---|---|
| Prompt 引导 | 通过 Prompt 提供软性指令 | 简单易用，零成本 | 无法保证 100% 格式合规，受模型随机性影响 |
| 输出后校验 | 生成后进行语法和内容验证并修正 | 提供事后保障，提高可靠性 | 需要额外开发和计算开销，不解决根本问题 |
| 约束解码 | 逐 token 通过外部语法规则硬性约束 | 100% 保证语法合规，从根本上解决非确定性 | 实现复杂，有性能开销 |
| 监督微调 | 在有标签数据集上训练 | 永久改变模型行为，无需复杂 Prompt | 存在 plateau 现象，需要高质量标注数据 |
| RL 强化 | 通过奖励机制塑造输出 | 能突破 SFT 高原，适合多步推理任务 | 成本高，训练不稳定 |
| 接口化能力 | 约束解码等能力抽象为 API | 简化开发流程，端到端类型安全 | 依赖特定模型供应商，可控性有限 |

技术选型时不应盲目追随最新方案，而应综合比较合规性要求、开发成本和供应商锁定风险。

对应证据：`agent_img_009_010_1123d496a7e0`、`agent_img_009_009_fd719d4b2dcf`。

### 金融场景三层防幻觉体系

金融 RAG 场景需要事前、事中、事后三层防线：

1. **事前 — 结构化输出约束**：强制模型输出 JSON 格式，每个字段（利率、条款编号、结论）有严格类型和值域约束，数字字段直接从 RAG 原文抽取而非推算。这从根源上杜绝推理型幻觉。
2. **事中 — 引用一致性自动校验**：对模型输出中每一句引用声明，用文本相似度或 NLI 模型自动比对原始检索文档，做声明级溯源验证。蚂蚁集团 HOP 框架明确提出此机制。
3. **事后 — 置信度分级 + 人工审核路由**：低置信度或校验失败的案例强制路由至人工复核，同时沉淀为难例数据集用于后续模型迭代。

对应证据：`agent_img_001_017_29006a1055a1`。

## Prompt 架构与缓存

### 系统 Prompt 段落注册表

系统 Prompt 不应是一个巨大字符串，而应拆成有名称、可定位、可标注缓存属性的段落注册表。

- **稳定前缀**：Agent 身份、安全规则、工具使用原则、输出格式、错误处理规则。
- **动态边界后**：当前时间、用户状态、会话变量、检索结果、工具调用结果。

Memory 注入应放在 system prompt 前部（如 Claude Code 的第十段，CLAUDE.md 之前），在 REPL 启动时注入。

对应证据：`agent_img_010_027_8cb3c4541134`（归入 Agent 工程模式审校）、`agent_img_010_026_92e884da861b`（归入 Agent 工程模式审校）、`agent_img_010_023_d582b70d06a2`。

### 工具调用边界

Prompt 必须明确约束工具调用边界。很多 Agent 的 System Prompt 只写了"你可以使用以下工具……请正确调用工具"，但没有定义：
- 什么情况下应该用哪个工具。
- 什么情况下应该拒绝调用。
- 工具调用失败后如何处理。

约束不足会导致模型在不确定场景下随意调用工具，产生难以排查的错误。

对应证据：`agent_img_007_001_e34249d9df55`。

### RAG 专用 Prompt 结构

RAG 场景需要有专用的 Prompt 结构设计：
- System Prompt：角色定义 + 参考文档说明 + 回答规则。
- User Prompt：格式化检索片段 + 具体问题。

两者分离可以让 System Prompt 保持稳定（可缓存），User Prompt 随每次检索结果动态变化。

对应证据：`agent_img_001_019_e69b9d279a6c`。

## 提示词与运行时双层防御

Agent 关键约束应实现双层防御而不是只靠 prompt：

- **提示词层**：告诉模型必须做什么（如"编辑前先读取文件"）。
- **运行时层**：工具执行时真正检查前提条件是否满足（如检查对话历史中有没有 Read 调用）。

单靠提示词的防御是软约束，模型可能忽略；单靠运行时校验则模型可能根本不尝试。双重覆盖才能可靠。

对应证据：`agent_img_010_031_94b80bbf7ca5`。

## AskUserQuestion 工具演进

Claude Code 团队在处理"任务中途询问用户"这个需求时，经历了三次迭代：

1. **第一版**：给已有工具（如 Bash）加参数来顺带提问 — 模型难以学会这种用法。
2. **第二版**：Prompt 里告诉模型"需要确认时输出特定格式" — 容易被忽略。
3. **第三版**：独立的 `AskUserQuestion` 工具，让提问成为一等工具调用 — 模型自然学会在需要时暂停。

结论：重要的交互模式应作为一等工具，而不是嵌入其他工具或依赖 prompt 中的行为指导。

对应证据：`agent_img_010_005_772065f10bc8`。

## 长上下文处理

### Map-Reduce 长上下文评审

当单次评审的上下文（大 MR 或大规则集）超出模型注意力有效范围时，可采用 Map-Reduce 模式：

1. **拆分**：将长上下文分解为多个小段。
2. **处理**：各段独立评审。
3. **合并**：汇总各段评审结果。

这本质上是用分布式处理逻辑缓解单次上下文窗口的注意力衰减问题。

对应证据：`agent_img_009_003_61e2922ef802`。

## MCP 架构

### 三层结构

MCP 不应被简单理解为 Client + Server 的二元结构：

- **角色层**：Host、Client、Server — 谁在通信，角色定位、会话、权限边界。
- **能力层**：Tools、Resources、Prompts — 暴露什么能力，三层独立演进。
- **协议层**：stdio / Streamable HTTP — 消息怎么传，消息格式、传输方式。

忽略 Host 角色是常见踩坑点，Host 负责管理 Client 生命周期和权限边界。

对应证据：`agent_img_002_018_21ba6bfa8425`、`agent_img_002_019_6a3f08dd5a27`。

## Hooks 机制

Claude Code 当前支持的 Hook 点覆盖了工具执行生命周期的关键节点：
- PreToolUse、PostToolUse、PostToolUseFailure — 工具执行前后。
- Notification — 通知事件。
- 用户提交前/后等交互节点。

Hooks 适合做轻量阻断和自动化，例如在敏感操作前插确认、在操作后自动记录日志。

对应证据：`agent_img_010_006_91f3a9315fd6`。
