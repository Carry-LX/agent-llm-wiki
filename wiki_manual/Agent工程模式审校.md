# Agent 工程模式审校

这页整理 Agent 系统架构中除 Memory 外的工程模式。目标是把自动页里受 OCR 影响的零散摘录，整理成可复用的工程判断。

## 工具安全

Agent 工具系统应采用安全默认值。工具构建时默认假设最危险的情况，必须由工具开发者显式声明并发安全、只读等属性。这会把安全从“默认允许”转成“显式退出风险”。

工具属性不应只由全局工具名决定，而应感知输入。例如同一个 `BashTool`，`ls` 和 `rm` 的安全属性完全不同。细粒度的输入感知是精确权限控制的基础。

对应证据：`agent_img_010_004_7748829a7526`。

## 文件编辑硬约束

文件编辑工具应强制“编辑前必须读取文件”。这不是提示词建议，而是运行时硬约束：工具执行时检查对话历史里是否存在对应 `Read` 调用；如果没有，编辑工具直接报错。

这样做是为了避免模型凭记忆或猜测修改文件。文件可能已经被用户、其他 Agent 或外部程序改过，直接编辑会把旧状态写回去。

对应证据：`agent_img_010_030_907dbaa011f8`。

## Prompt 架构与缓存

系统 Prompt 不应是一个巨大字符串，而应拆成有名称、可定位、可标注缓存属性的段落注册表。

适合放在稳定前缀里的内容：

- Agent 身份。
- 安全规则。
- 工具使用原则。
- 输出格式。
- 错误处理规则。

应放在动态边界之后的内容：

- 当前时间。
- 用户状态。
- 会话变量。
- 当前任务上下文。
- 检索结果。
- 工具调用结果。

这样既能提升 prompt caching 命中率，也让后续调试和替换单个提示段更容易。

对应证据：`agent_img_010_027_8cb3c4541134`、`agent_img_010_026_92e884da861b`。

## 工具检索字段

工具元信息应区分 `searchHint` 和 `description`：

- `searchHint`：给 ToolSearch 匹配用的短能力标签。
- `description`：工具已加载后，给模型理解参数、调用边界和行为约束用的正式说明。

这种拆分能降低工具检索阶段的 token 成本，也避免把长说明全部塞进初始上下文。

对应证据：`agent_img_010_003_df2c941682fd`。

## 重试与错误处理

Agent 重试机制不能只写“失败就重试”。稳妥设计应包含：

- 预算：例如总重试次数、特定错误码单独次数、等待上限。
- 分类：429、529、408、409、401、403、5xx、连接错误的语义不同。
- 过载降载：过载时应减少请求，而不是所有请求一起重试。
- 幂等性：涉及写文件、调用工具或提交变更时，要考虑重复执行副作用。
- 可观测性：每次 query、success、error 都打事件，方便线上定位。
- 用户体验：等待期间让用户看到状态，而不是像程序卡死。

指数退避还应加入随机抖动，避免大量客户端在固定时间点同时重试，进一步放大后端压力。

对应证据：`agent_img_010_029_ea01ad355cb5`、`agent_img_010_028_deae5c10a6f8`。

## Workflow 与 Skill 编排

Workflow 可以封装业务相关上下文和 Skills 执行编排。复杂任务不是只调用单个 Skill，而是按步骤串联多个 Skill，并把业务上下文作为编排的一部分。

一个标准 Agent 轨迹应包含：

- 用户问题。
- 模型思考。
- 工具调用。
- 工具观察结果。
- 最终回答。

工具返回内容属于外部 observation，不应和模型内部推理或最终回答混在一起。

对应证据：`agent_img_001_015_5be324090a5c`、`agent_img_005_001_bad29a1acc99`。

## 多 Agent 编排

多 Agent 编排常见模式包括：

| 模式 | 适合场景 | 主要风险 |
|---|---|---|
| 顺序管道 | 强依赖步骤 | 前序错误会传递 |
| Map-Reduce | 并行任务 | 汇总和冲突裁定困难 |
| 层级嵌套 | 深层复杂任务 | 需要限制深度 |
| 路由分发 | 请求类型多样 | 路由错误会放大 |

## 上下文压缩

长上下文清理可以采用“头部保护、尾部保护、中间摘要”策略：

- 头部保护：保留系统指令和初始任务定义。
- 尾部保护：保留最近几轮对话，维持短期连贯性。
- 中间摘要：对冗长工具调用过程和推理步骤裁剪，并生成摘要替代原始细节。

## Badcase 闭环

Agent 系统上线后的 badcase 闭环应包含收集、分类、修复验证、全量回归和灰度发布。

收集渠道可以包括：

- 用户反馈。
- 客服工单。
- 自动检测。

分类维度可以包括：

- 检索失败。
- 幻觉生成。
- 路由错误。
- 知识缺失。

每次修复后，应先在原 badcase 上验证，再跑全量回归，确认核心指标没有明显退化，最后灰度发布。

对应证据：`agent_img_001_005_ce1638579d53`、`agent_link_001_004_b0fd20db1549`。
