# Text2SQL 与查询校验审校

这页是对 `wiki_generated/Text2SQL_与查询校验.md` 的人工审校版。自动页中的证据表仍保留原始回溯价值，但摘要区曾混入 OCR 噪声和两条 RAG/Milvus 性能材料；本页以 `data/claim_overrides.json` 中的审校 claim 为准。

## 核心结论

Text2SQL 在 Agent 系统中不应被设计成自由执行器，而应是一个只读、受控、可校验的工具。Agent 负责意图判断、歧义澄清和流程调度，Text2SQL 只在条件齐全时生成并执行查询。

Text2SQL 的风险不止是“SQL 写错”。更现实的问题包括：查询越权、危险关键字、结果缺少用户要求的关键实体、数值异常但语法正确、以及历史分布明显不合理。因此执行前和执行后都要校验。

## 执行前校验

执行前至少检查：

- SQL 是否只读，例如只允许 `SELECT`。
- 表和字段是否在授权范围内。
- 是否包含危险关键字。
- 是否自动加上 `LIMIT`，避免无界查询。
- 用户问题是否缺少必要槽位；缺槽时先澄清，不直接查询。

## 执行后校验

执行后不能只把结果交给模型回答，还要检查：

- 结果是否为空。
- 数值是否落在合理范围内。
- 查询结果是否覆盖用户问题中的关键实体。
- 是否和历史分布明显冲突。

例如证据图中，SQL 返回了全国销售额 `1234万`，但用户问的是“华东区”；执行后校验发现结果缺少“华东区”实体，并且数值与历史华东区平均值偏离过大，因此应返回澄清提示，而不是把查询结果包装成答案。

## 数据构造与训练

Text2SQL 训练数据可以从问答模板开始，通过变量随机填充生成基础样本，再用大模型改写问题表达，扩充自然语言问法。生成 SQL 时必须约束表名和字段名来自已知 schema，并处理中文数字到阿拉伯数字的转换。

双向增强也是一个可用方向：从问题生成 SQL 是正向构造，从 SQL 反推问题是逆向构造。当前材料记录的实验中，双向增强框架相对少量人工标注 LoRA 提升执行准确率 `16.3%`，相对 few-shot 提示学习提升 `35.7%`。
