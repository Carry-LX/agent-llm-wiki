# TMIC AI 小新：Agent 设计方法论

> 阅读淘宝 TMIC AI 小新 Agent 设计思路后的体系化总结

---

## 基石信念：上下文工程决定一切

全文所有设计决策都建立在一个判断上：

> **当前基模的推理能力已经足够强了。Agent 失败，绝大多数时候不是因为模型"笨"，而是因为它看到的上下文不对——要么信息太多淹没了关键信号，要么信息不够导致判断依据缺失。**

所以整个团队的工程精力不是花在"怎么写更好的 Prompt"，而是花在"如何让模型在正确的时刻只看到正确的信息"。所有组件都是围绕这一个目标服务的。

---

## 第一层：私有知识的两种注入方式——Skills vs RAG

业务场景下，模型缺的不是通用推理能力，缺的是**这个业务的领域知识**。这部分知识分两类，用两种不同的机制注入：

| | Skills | RAG |
|---|---|---|
| **是什么** | 手写的业务规则文档 | 历史案例检索 |
| **内容举例** | "没指定价格带时默认取GMV最高的""类目获取不到时用主营类目兜底" | "三个月前有人问'潜力最高的类目'，实际分析路径是筛选竞争强度最低的叶子类目" |
| **覆盖范围** | 通用规则，整个业务模块的人问都适用 | 具体案例，跟你这次提问最相似的历史经验 |
| **本质** | 约束行为边界 | 提供参考模板 |

两者互补使用：Skills 管的是"不管谁来问，在这个业务里你都应该这么干"；RAG 管的是"跟你类似的问题，别人是这么一步步走通的"。

---

## 第二层：用小步骤换来高密度上下文

这是被很多 Agent 设计忽略的一点：**上下文不是越多越好。** 把全量工具、全量规则、全量案例全塞给 Planning，只会让它决策更慢、更容易出错。

文章的做法是：在正式 Agent 启动前，插入一个轻量的"业务模块识别"步骤。这个小步骤做的事情很简单——判断用户问题属于哪几个业务模块——但它带来了一连串的收益：

- 工具集从"全量（几十上百个）"变成"相关子集（10-20个）"
- Skills 只注入相关模块的，不相关的不会干扰 Planning
- RAG 只检索相关模块的案例，不会召回无关内容

> **引入这个小步骤本身有成本**——它多了一次调用，而且识别错了后面全错。但团队判断这个成本是可接受的，因为它是一个边界清晰、可评测、可针对性优化的分类任务。用可控的风险换来了整个上下文环境的信息贴合度提升。

这个模式在 coding agent 场景下完全适用——先判断涉及哪个项目、加载对应的 CLAUDE.md、读取相关文件，再进入真正的编码推理。

---

## 第三层：能前置的全部前置

识别完业务模块后，顺手把后续可能用到的全局参数一次性并行取回来：userId、主营类目、时间范围、新品列表前几条……

> **动机很简单**：不要让 Agent 在执行过程中为了拿一个通用参数多走一轮完整的 LLM 往返。能前置的就前置。数据量大的做智能截断（前5条加载，其余提供接口入口按需调用）。

本质上这也是在减少 Agent 执行时的"非核心决策"——常用的东西提前备好，Agent 把精力留给真正需要推理的决策。

---

## 第四层：多轮 Agent 的方向感——TodoList

**这是一个不下场做工程很难体会到的痛点。**

标准 ReAct 每一步只看当前状态，没有全局视角。面对 5-10 步的复杂分析任务，Agent 走到第 4 步可能已经忘了第 1 步为什么要取那个数据，方向感逐渐丢失。

> **TodoList 就是一个锚**：第一轮 Agent 先生成完整的执行计划，之后每一轮都带着这个计划思考。它不改变 ReAct 的核心循环，只是在每次"Thought"的时候多挂了一个全局参考系——"我现在在计划的哪个位置？还剩什么没做？最初的目标是什么？"

这是针对多轮 Agent 的一个结构性补丁，解决的是 ReAct 原生模式的天生缺陷。

---

## 第五层：Tree Action——把编排逻辑从大模型身上剥离

**这是全文最有原创性的一个设计。**

问题的起点是：Agent 经常要执行有依赖关系的工具链（A的输出是B的输入 → B的输出是C的输入），标准 ReAct 下每步都是一次大模型往返，3步就是3次完整推理。大量时间是重复的——Agent 在第2轮看到A的结果后判断"下一步该调B"，这个判断在第1轮其实就能做出来。

> **Tree Action 的做法**：让 Planning 在单轮中输出一组带依赖关系的 Action（"Action 2 依赖 Action 1 的输出"），系统按层级自动执行。全部完成后一次性把结果还给 Planning。

**关键细节：依赖之间的参数传递用小模型。** A返回了 `{cateId: 1801005}`，B需要入参 `cateId`——这个映射不需要大模型来推理，一个小模型完全够用。所以 Tree Action 的本质分工是：

- **大模型**负责规划："先做A再做B，B依赖A的输出"——这是需要全局理解能力的
- **小模型**负责串联："把A输出的cateId填到B的cateId入参"——这是机械映射
- **系统**负责按层级调度执行——这是确定性的编排逻辑

实测效果：Planning 平均次数从 3.20 降到 2.44。省掉的不是执行时间，是**大模型重复推理的次数**。

这个模式其实已经带有 SubAgent 的味道了——Tree Action 内部自动串联执行的这套逻辑，相当于一个轻量级的"执行子Agent"，只是它不做推理、不做规划，只负责按图执行。

## 总骨架：一个可迁移的 Agent 设计框架

```text
┌─────────────────────────────────────────────────────┐
│ 阶段1: 上下文加载                                     │
│ 业务场景识别 → 过滤工具/Skills/RAG → 预取全局参数          │
│ 目标：让后续 Agent 只看到高度贴合当前任务的信息              │
├─────────────────────────────────────────────────────┤
│ 阶段2: 规划 (Planning)                                │
│ 首次：生成 TodoList（全局执行计划）                       │
│ 每轮：带着 TodoList 决定下一步（Thought → Action）          │
│ 目标：多轮 Agent 不丢失方向感                            │
├─────────────────────────────────────────────────────┤
│ 阶段3: 执行 (Tree Action + SubAgent)                   │
│ Tree Action：一轮输出多Action（带依赖关系），系统按层级执行     │
│   依赖串联用小模型                                        │
│ SubAgent：独立子任务并行委托，上下文隔离                      │
│ 目标：减少大模型往返次数，简单编排逻辑降级                     │
├─────────────────────────────────────────────────────┤
│ 阶段4: 分析与输出 (Analysis 独立节点)                     │
│ 分离出独立的分析节点，支持流式输出                            │
│ 目标：不阻塞主流程，用户体验友好                            │
└─────────────────────────────────────────────────────┘
```

核心原则就一句：

> **相信基模的推理能力，把全部工程精力投入到"让基模在每一刻看到的上下文是对的、够的、不冗余的"。**

其他所有组件——Skills、RAG、业务识别、TodoList、Tree Action、SubAgent、异步 Summary——都是这句话的不同实现方式。

## 深度感悟：SubAgent 优化的本质是"角色定义"，不是"性能调优"

大多数人看到"SubAgent 从 103 秒优化到 20 秒"，第一反应是：做了什么技术优化？是不是换了更快的模型？是不是并行化了？是不是缓存了？

但真正的优化手段是什么？**砍掉 TodoList，砍掉 Analysis 详细输出。** 这不是性能调优，这是**纠正一个角色定位错误**。

### 核心认知：SubAgent ≠ 小型 Agent

初版设计的思维惯性是：SubAgent 就是 Agent 的缩小版。既然 Agent 有 TodoList、有 Planning、有 Analysis，那 SubAgent 也应该有，只是"缩小一点"。于是 SubAgent 被设计成"完整 Agent 减去 Human 调用和 SubAgent 调用"。

但这完全搞错了。Agent 和 SubAgent 的关系不是"大版本 vs 小版本"，而是**"规划者 vs 执行者"**。它们不是同一角色的不同尺寸，而是**不同角色**。

### 从角色推导约束，而不是从功能罗列例外

**① "它拿到的是已分解后的任务，所以不应该再做分解"**

这句话对应砍掉 TodoList。不是"TodoList 太重了所以优化掉"——这是工程师思维。真正的逻辑是：**如果你的输入已经是分解后的原子任务，你就不应该拥有分解能力。** 这不仅仅是性能问题，而是职责越界。SubAgent 如果有 TodoList，它可能会把"获取类目 1801 的价格带数据"再拆成"先获取价格带列表 → 再筛选 → 再排序"，这不仅是浪费，而且可能拆错。

**② "它不应该再衍生子 Agent，因为任务本身已分解到位"**

这句话对应禁止 SubAgent 调用 SubAgent。这不是"加一条限制规则"那么简单。这背后的原则是：**分解权归属于规划层，不在执行层。** 如果执行层也可以分解，就会产生递归分解——主 Agent 拆给 SubAgent-1，SubAgent-1 又拆给 SubAgent-1-1，SubAgent-1-1 又拆给 SubAgent-1-1-1……每一层都可能产生理解偏差的累积，而且主 Agent 完全失控。主 Agent 只知道"我把任务交给 SubAgent-1 了"，但不知道 SubAgent-1 在下面又搞了一层嵌套，出了问题无法追溯。

**③ 总结起来一句话：SubAgent 的能力集合，应该由它的角色定义来推导，而不是从 Agent 的能力集合做减法。**

不是"Agent 有 10 个能力，SubAgent 减掉 2 个不合适的能力剩下 8 个"。而是"SubAgent 的角色是执行者，执行者天然只需要 3 个能力：理解任务、调工具、返回结果"。其余 7 个——规划、分解、衍生子代理、询问用户、生成分析报告……——不是"被砍掉"，而是**本来就不属于这个角色**。

### 从"做减法"到"做定义"

这篇文章最有价值的地方，是它展示了一条关键的认知升级路径：

| 阶段 | 思维模式 | 行为 |
|------|---------|------|
| 初版设计 | "Agent 缩小就是 SubAgent" | 照搬 Agent 的全部能力，加两条限制 |
| 遇到问题 | 性能太差，103 秒 | — |
| 排查分析 | 时间耗在 TodoList 和 Analysis | — |
| 表层解法 | "砍掉这两个慢的部分" | 删除 TodoList 和详细 Analysis |
| **深层认知** | **SubAgent 不是缩小版 Agent，是执行者** | **从角色定义倒推能力边界** |

表面上做的事情一样——都是删除 TodoList 和 Analysis。但认知层次完全不同：

- "因为慢所以砍" → 下次遇到另一个慢的操作还会砍，直到把 SubAgent 砍成一个空壳
- "因为不属于执行者角色所以不该有" → 不仅砍得对，而且**新增能力时也有判断标准**——"这个能力属于规划者还是执行者？"

这就是为什么这篇文章值得认真读。它不是炫技，而是在展示一个真实团队在真实业务压力下，怎么一步步想清楚一个概念。从 103 秒到 20 秒，优化的是代码，但真正被优化的是**对"SubAgent 到底是什么"的理解**。

## 深度感悟：从 Summary 到评估闭环——DeepAgent 的"基础设施层"是怎样一层层搭起来的

回头看技术点六到十二，它们表面上讲的是七个独立组件，但串起来看，其实是在讲同一件事：**如何给 AI 的自主决策搭建一个可持续运转的基础设施。**

### 第一层认知：上下文管理不是"清垃圾"，是"做编辑"

Summary 和异步 Summary 看起来是工程优化——压缩上下文、异步执行、省 token、省时间。但如果只看到这一层，就没抓住重点。

Summary 真正的设计思想是：**上下文不是"存储空间"，是"工作台面"。** 存储空间的管理策略是"满了就扔旧的"（滑动窗口），但工作台面的管理策略是"跟当前任务无关的收起来、随时可能用到的留在手边"。

这就是为什么 Summary 的压缩策略不是"保留最近 N 轮"，而是**读 TodoList、读工具列表、判断贯穿性 vs 阶段性**。Summary 在做的事情本质上是一个编辑的工作——不是把所有草稿按时间堆成一摞，而是把已完成的章节归档、把正在写的摊开、把参考资料放在触手可及的位置。

异步化又往前推了一步：**编辑工作不应该打断写作过程。** 把 Summary 从串行改成异步，本质上是在说——整理工作台这件事，不需要作者停下笔来等。你在旁边悄悄整理，作者继续写，整理完了把干净台面推过去就行。这是一个"基础设施不应该让使用者感知到它的存在"的设计原则。

### 第二层认知：职责分离不是为了"架构好看"

Analysis 独立和 FileSystem 放弃，放一起看特别有意思——一个是"拆出来"，一个是"不引入"。两件事方向相反，但判断标准完全一致：**这个职责属于谁？**

Analysis 为什么要从 Planning 中拆出来？不是 Planning 写不好报告，而是"决策调度"和"内容生成"是两种完全不同的工作模式：
- 决策调度需要的是"快速、准确、不被细节淹没"
- 内容生成需要的是"完整、深度的上下文 + 流式输出"

让一个人同时做这两件事，他只能折中——决策时被报告细节拖慢，生成报告时被决策结构约束。拆开之后，两个角色各自在自己的最优条件下工作。

FileSystem 为什么不引入？不是它不好，而是**它要解决的问题在这个场景里不存在。** 工具返回的都是有用的数据，Summary 已经在做压缩了——FileSystem 的"卸载再读取"变成了纯开销。

这两件事合在一起说明一个原则：**组件的存在理由不是"这个技术很先进"，而是"这个职责缺人干"。** 反过来，如果一个职责已经有人在干了，再加一个人就是添乱。

### 第三层认知：RAG 是"软约束"，不是"知识库"

在 DeepAgent 这个上下文里，RAG 扮演的角色远比"知识库"重得多。

回到那个核心困境：**DeepAgent 把决策权给了 AI，当 AI 跑偏时你怎么办？** 改 Prompt？Prompt 只能写通用规则，管不了具体场景的个性化需求。改 Skills？Skills 也是通用规则，而且写得越细越死板。这两个手段都太"硬"了。

RAG 提供的是第三种手段——**软约束。** 不是告诉 AI"你必须这么做"，而是告诉它"历史上类似的问题，我们都是这么做的"。AI 仍然有自主决策权，但它做决策时的起点不是"一张白纸"，而是"一个被验证过有效的模板"。这不是剥夺自由，而是**减少从零规划的概率，从而减少发散的概率。**

更深一层：RAG 是 DeepAgent 模式下的**干预接口**。在 Workflow 时代，改代码就是干预。在 DeepAgent 时代，代码是通用的、路径是 Planning 生成的——你没法改代码来修一个具体的分析路径问题。能改的只有 Prompt、Skills 和 RAG。而 Prompt 和 Skills 改起来影响面太大（改一条规则可能波及其他所有模块），RAG 的影响是**局部的、可定向的**——你只需要给这个具体问题补充一条相似的参考案例，Planning 下次就会往这个方向靠。

这就是为什么团队说 RAG 是"最重要且最有效的纠偏手段"——不是因为它功能最强，而是因为它是**唯一一个可以精准干预而不产生副作用的入口。**

### 第四层认知：评估闭环是 DeepAgent 的"免疫系统"

这是整篇文章最被低估的设计，但它可能是最重要的。

为什么？因为 DeepAgent 和定制化 Workflow 有一个根本性的差异：**Workflow 上线后是"死的"，DeepAgent 上线后是"活的"。**

Workflow 的执行路径是写死的，上线之后只要不改代码，昨天的行为跟今天一模一样。你不需要"持续评估"——它是确定性的，测一次就够了。

但 DeepAgent 每天都在做新决策。改了 Prompt 一句话，所有问题的分析路径都可能微妙地漂移。加了 10 条 RAG，某些问题变好了，另一些可能被误导了。甚至大模型后台换了版本，行为也可能不一样。**DeepAgent 不是一个"部署完就完了"的系统，而是一个"部署完才开始变化"的系统。**

这种系统如果没有持续评估，就像一个人没有免疫系统——你不知道它什么时候变差了，等你发现的时候（用户投诉、数据出错）往往已经是严重问题。

团队设计的五步闭环（用例收集 → 批量执行 → 模型评测 → 人工复核 → 数据回流）本质上就是一套免疫系统：
- **用例收集**：建立"正常状态"的基线——这 150 个问题，预期应该怎么回答
- **批量执行 + 模型评测**：定期体检——每次改了什么之后，跑一遍看有没有退化
- **人工复核**：专家会诊——自动评测看不出来的问题，人来看
- **数据回流**：自我学习——把好的案例喂回 RAG，让系统越来越强

而数据回流（第 5 步）是最关键的一环。没有它，评估只是"发现问题"，不能"自动修复"。有了它，评估闭环就从"监测系统"升级成了"自进化系统"——评估发现的高质量案例自动入库为 RAG，下一次 Planning 就能参考，下一次评估就能验证。**RAG 的库越丰富，Planning 越稳；Planning 越稳，评估越高；评估越高，发现的好案例越多；好案例越多，RAG 库越丰富。** 这是一个正向飞轮。

而这一切都不需要改代码、不需要重训模型、不需要人工逐个编写分析规则。你只需要保证评估体系运转正常，系统自己会越来越好。

### 总览：这些技术点的本质关系

回到开头，这七个技术点的关系不是"一个架构的七个零件"，而是**三个层次**：

| 层次 | 包含 | 解决的问题 |
|------|------|-----------|
| **上下文工程** | Summary、异步 Summary、Analysis 独立、FileSystem 放弃 | 让 AI 在正确的时机拿到正确的信息，不被无关信息淹没 |
| **行为约束** | RAG、Skills | 让 AI 在自主决策时有一个"被验证过的起点"，不跑偏 |
| **持续进化** | 自动化评估闭环 + 数据回流 | 让系统上线后不是"维护"，而是"生长" |

第一层保证 AI 能干活，第二层保证 AI 干对活，第三层保证 AI 越干越好。

这三层缺一层都会出问题：只有上下文工程没有行为约束，AI 能干但会跑偏。只有行为约束没有上下文工程，AI 方向对了但被信息淹没。有前两层没有第三层，系统上线时是好的，但慢慢变差你也不知道。

这篇文章最让人印象深刻的地方在于：它不是讲一个聪明的一次性设计，而是讲一个团队怎么认识到"AI 自主系统必须有一套持续运转的基础设施"，然后一层层把这个基础设施搭起来。**代码让 AI 能跑，评估让 AI 能持续跑对，RAG 让 AI 能越跑越好。** 这三句话，比任何一个单独的组件设计都更重要。
